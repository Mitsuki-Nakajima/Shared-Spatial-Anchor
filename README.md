# Shared Spatial Anchor / Multiplayer Mixed Reality Prototype


# Overview

This project explores the implementation of a multiplayer mixed reality (MR) environment using Unity and Meta Quest 3.

The primary goal is to create a virtual version of the NYU Makerspace where:
- users physically inside the Makerspace and
- users joining remotely from different locations

can interact together inside the same shared virtual environment.

The main technical challenge explored in this project is:

> How can users in different physical environments share a common spatial coordinate system inside one virtual mixed reality space?

This report is written as both:
- a development report
- and a tutorial/documentation guide



# Initial Research and Template Exploration

At the beginning of the project, we explored Meta’s Multiplayer Mixed Reality template and Shared Spatial Anchor systems.

The template demonstrated:
- multiplayer networking
- avatar synchronization
- spatial alignment systems
- colocation workflows
- shared anchor concepts

This phase helped us understand how Meta structures multiplayer mixed reality applications before rebuilding a more lightweight and customizable version.



# Core Problem

The provided Meta template primarily assumes:
- users are located in the same physical room
- users share a common real-world coordinate system

However, our project requires:
- users joining from home
- users joining from different rooms or buildings
- remote participation in the same virtual Makerspace

This introduces a major challenge:

> Each headset has its own local XR tracking space, so users do not naturally share the same coordinate system.

The project therefore focuses not only on networking, but also on:
- spatial calibration
- coordinate system alignment
- shared virtual positioning

Instead of depending entirely on Meta’s template, the current architecture rebuilds only the essential systems needed for the project.



# Overall System Architecture

The system separates the application into two main spaces:

## 1. Local XR Space

Each user has their own local XR tracking space generated by the headset.

This includes:
- headset tracking
- controller/hand tracking
- room-relative movement

Each local XR space exists independently.



## 2. Shared Virtual Makerspace Space

A shared virtual `MakerspaceRoot` acts as the global coordinate system for the metaverse environment.

All player positions are converted into this shared coordinate system before being synchronized across the network.



# Overall Workflow

The intended workflow is:

1. The user’s headset and hands are tracked locally through `XR Origin`
2. Local poses are converted into shared Makerspace coordinates
3. The transformed poses are synchronized over the network
4. Other users render the avatar inside the same virtual Makerspace

This allows users in completely different physical locations to appear together in one virtual MR environment.

# Production-Grade Calibration System

One of the largest developments during this project was the evolution of the calibration system.

The original project used a very basic one-point calibration system that aligned:
- one local headset position
with
- one target virtual position

While simple, this method introduced several limitations:
- inaccurate rotation
- drift
- unstable alignment
- poor scaling for larger spaces

To improve this, the project evolved into a much more advanced multi-point calibration architecture.



# Enhanced Calibration Architecture

The calibration system now includes several modular components:

## CalibrationData

`CalibrationData`
- stores structured calibration information
- supports multiple calibration points
- stores confidence scores
- stores quality metrics and metadata

The system is designed to support future versioning and extensibility.



## CalibrationManager

`CalibrationManager`
- manages the calibration pipeline
- supports both legacy and enhanced calibration
- computes rigid transformations
- stores calibration persistently
- validates calibration quality
- provides event-driven architecture for integration

The system maintains backward compatibility with the original single-point calibration approach.



## CalibrationValidator

`CalibrationValidator`
- evaluates calibration quality
- checks average and maximum error
- detects degenerate calibration configurations
- supports configurable validation thresholds

This helps prevent invalid or unstable calibration results.



## CalibrationUI

`CalibrationUI`
- provides guided calibration flow
- supports visual feedback
- provides progress tracking
- includes retry functionality
- supports real-time stability monitoring

The goal is to make the calibration workflow easier and more user-friendly.



## CalibrationPointVisual

`CalibrationPointVisual`
- visually represents calibration points
- displays calibration state
- supports confidence-based feedback
- includes pulsing and visual indicators

This improves usability during the calibration process.



## NetworkCalibrationSync

`NetworkCalibrationSync`
- synchronizes calibration across multiple users
- supports host-controlled calibration
- synchronizes calibration for late-joining users
- manages calibration versioning

This ensures all users operate using consistent shared-space transformations.



# Multi-Point Calibration Workflow

The updated workflow is:

1. The user collects several calibration points
2. Each local XR point is paired with a corresponding target Makerspace point
3. The system computes an optimal rigid transformation
4. The resulting transformation matrix aligns the user’s local XR space with the shared virtual environment

This approach is significantly more accurate than the original one-point alignment method.



# Kabsch Algorithm Implementation

To improve spatial alignment accuracy, the project experiments with a simplified implementation of the Kabsch algorithm.

The Kabsch algorithm computes:
- the optimal rotation
- and translation

between two sets of 3D points.

In this project, it is used to align:
- local XR calibration points
with
- target Makerspace calibration points

This allows the system to estimate a more accurate shared-space transformation matrix.



# Calibration Validation

The enhanced system also includes:
- residual error calculation
- average error measurement
- maximum error measurement
- threshold-based validation
- degenerate configuration detection

Examples of invalid configurations include:
- points too close together
- collinear point arrangements
- unstable transformations

This allows the system to reject unreliable calibration solutions.



# Calibration Persistence

The calibration system supports persistence using `PlayerPrefs`.

This allows:
- calibration data to survive application restarts
- reduced setup time for repeated sessions

Calibration data is serialized and stored locally.



# SharedSpaceManager

`SharedSpaceManager` handles:
- shared virtual coordinate systems
- local-to-shared pose conversion
- alignment matrix application

This acts as the bridge between:
- local XR tracking space
and
- shared virtual Makerspace space



# PlayerRigTracker

`PlayerRigTracker`
- reads local XR transforms
- applies calibration transformations
- synchronizes avatar transforms
- monitors calibration status
- supports fallback calibration systems
- supports network synchronization

The script also includes:
- update-rate limiting
- calibration warnings
- fallback behaviors
- event subscriptions



# Fallback Calibration Architecture

The current calibration pipeline supports layered fallback behavior:

```text
Enhanced Multi-Point Calibration
        ↓
Legacy Single-Point Calibration
        ↓
SharedSpaceManager Conversion
        ↓
Raw Local Pose
```

This architecture improves:
- robustness
- debugging
- testing flexibility
- failure recovery

Even if one calibration system fails, the project can continue functioning using fallback approaches.



# Tutorial: Project Setup

## 1. Create a Unity Project

Create a new Unity 3D project using:
- Unity 6 or compatible version
- Android build support

Target device:
- Meta Quest 3



## 2. Install Required Packages

Install:
- XR Plug-in Management
- OpenXR
- XR Interaction Toolkit
- Unity Netcode for GameObjects
- Unity Transport



## 3. Enable XR

In Unity:

```text
Edit > Project Settings > XR Plug-in Management
```

Enable:
- OpenXR for Android
- Meta Quest support



## 4. Add XR Origin

Create:

```text
GameObject > XR > XR Origin (VR)
```

This creates:
- headset tracking
- left hand tracking
- right hand tracking



## 5. Add Networking

Create:

```text
NetworkManager
```

Add:
- `NetworkManager`
- `Unity Transport`
- `SessionBootstrap`

Create Host and Client buttons for multiplayer startup.



## 6. Create Shared Makerspace Root

Create:

```text
MakerspaceRoot
```

This object acts as the shared virtual coordinate system.

Placeholder geometry such as:
- cubes
- planes
- walls

can be used during testing.



## 7. Create PlayerAvatar Prefab

Create:

```text
PlayerAvatar
```

Add:
- Head
- LeftHand
- RightHand

Add components:
- `NetworkObject`
- `NetworkTransform`
- `PlayerRigTracker`

Convert the object into a prefab and assign it to:
- `NetworkManager > Player Prefab`



## 8. Setup Calibration Systems

Create:
- `SharedSpaceManager`
- `CalibrationManager`

Assign:
- calibration data assets
- validation systems
- calibration prefabs
- MakerspaceRoot

The project also includes setup utilities for automated calibration system configuration.



## 9. Run the Project

To test:
1. Press Play
2. Start Host
3. Confirm that `PlayerAvatar(Clone)` appears
4. Test synchronization and calibration behavior

# Known Problems and Challenges

## 1. Remote Shared Space Alignment

The largest challenge remains:
- aligning users in different physical environments

Unlike same-room MR systems, remote users cannot rely purely on shared physical anchors.



## 2. Calibration Complexity

The enhanced calibration framework improves flexibility and accuracy, but also introduces:
- implementation complexity
- debugging difficulty
- dependency management concerns
- possible numerical instability



## 3. Numerical Stability

The current prototype includes a simplified Kabsch/SVD-style implementation.

Additional testing is still needed to verify:
- stability
- robustness
- edge-case behavior



## 4. Prototype-Level Environment

The project currently prioritizes:
- architecture
- networking
- calibration systems
- synchronization research

rather than:
- polished visuals
- production-ready interactions
